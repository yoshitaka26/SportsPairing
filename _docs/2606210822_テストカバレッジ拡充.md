# テストカバレッジの拡充（37 → 92 テスト）

## 概要
横展開で継続利用するライブラリの信頼性を担保するため、テストを **37 件から 92 件** へ拡充した。
モデル層・コスト計算の核（飽和曲線）・各種境界値・3 対 3 形式・基盤型（乱数器/エラー/設定）など、
既存テストで手薄だった領域を体系的に埋めた。新規テストファイルを 5 つ追加（既存ファイルは温存）。

## 背景
SportsPairing は複数アプリへ横展開し、重みアルゴリズムを継続的に育てる前提のライブラリ。
他アプリが本パッケージにバージョン固定で依存するため、**回帰の検出器＝テスト**が品質の生命線になる。
既存の 37 テストは主要なハッピーパス（基本マッチメイク・公平性・固定ペア・総当たり表など）を
押さえていたが、以下が手薄だった。

- **モデル層の契約**（`Match.summary` の整形、`Team.averageSkill` の nil 条件、`Player` の ID 等価性、`NumberedTeam` の正規化）
- **コスト計算の核**（飽和関数 `n/(n+1)`、スキル未設定時の評価スキップ、**対戦**重複の回避挙動）
- **エッジケース**（`makeRound` のコート 0 / 不正固定ペア / コート過多、`apply` の累積・試合外不変、各種クランプ）
- **境界値**（総当たり表の最小/最大人数、`maxMatches` の 0/負/超過、並び順の公平性）
- **多人数形式 3 対 3**（README で謳っているのにテストが存在しなかった＝最大の穴）
- **基盤型**（`SeededRandomNumberGenerator`、`MatchmakingError`、プリセット、`version`）

## 対応内容
`Tests/SportsPairingTests/` に 5 ファイルを追加。

| ファイル | 対象 | 主なテスト |
| --- | --- | --- |
| `ModelTests.swift` | Player / Team / Match / FixedPair / MatchFormat / Side | ID 等価性、drive/back フォールバック、averageSkill の nil 条件、summary 整形、teamSize クランプ、assignsSides 判定 |
| `ScoringTests.swift` | MatchScorer + 選出挙動 | 飽和関数の既知値（0→0, 1→0.5, 2→2/3, 3→0.75）・単調増加・1 未満飽和、スキル nil で skillGap=nil、対戦履歴の少ない新鮮な組を選ぶ |
| `EngineEdgeTests.swift` | Matchmaker エッジ + 3 対 3 | makeRound(コート 0／不正固定ペア→空／コート過多)、apply(二重適用の累積／試合外プレイヤー不変)、3 対 3 の生成・apply・公平性 |
| `ConfigRandomErrorTests.swift` | Options / Weights / Error / RNG / version | skillRange クランプ、プリセット値、Error の Equatable と description、乱数器の決定論・シード差・seed 0 非縮退、version |
| `RoundRobinEdgeTests.swift` | RoundRobinGenerator 境界 + 背番号モデル | 人数レンジ、最小人数ちょうど（ダブルス 4→3 / シングルス 2→1）、範囲外エラー、maxMatches 境界、先頭 2 試合で全員カバー、NumberedTeam/Match の正規化 |

合計 **55 件を追加**し、`swift test` で **92 件全通過**（約 1 秒）。

## 設計判断
- **既存ファイルを変更せず新規ファイルを追加**：差分を加算的に保ち、レビューと回帰追跡を容易にした。
  既存テストの意図を壊すリスクを避ける狙い。
- **内部実装も `@testable import` で直接検証**：飽和関数 `MatchScorer.saturating` や
  `MatchFormat.assignsSides`、`Side.conflicts` は internal だが、アルゴリズムの核なので
  「公開 API 経由の間接検証」ではなく**値を直接固定**した。曲線がずれたら即座に落ちる。
- **挙動テストは性質ベースで決定論に**：乱数が絡む箇所はシードを固定し、さらに
  「最小コスト候補が一意に定まる」入力を組んで、シードに依存せず必ず同じ結論になるよう設計した
  （例: 対戦回避テストは P5/P6 を完全に新鮮にして、最小コストの 4 人組が必ず両者を含むようにした）。
- **重い生成は避ける**：総当たり表の並び替えは候補数の二乗オーダー。最大人数（ダブルス 16 人 ≈ 5460 カード）の
  実生成はテストを遅くするため、上限値は**プロパティで検証**し、実生成は最小人数と 8 人までに留めた。

## 技術的ポイント
- **`n/(n+1)` 飽和曲線が標準化の要**：このライブラリの「重み = 項目間の相対重要度」という設計は、
  各コスト項目が 0〜1 に正規化されている前提で成り立つ。飽和関数の既知値テストは、その前提が
  将来の改修で崩れないための"防波堤"。`saturating(2) == 2/3` は浮動小数のため許容誤差で比較している。
- **`makeRound` は throw ではなく空配列で失敗を表す**：`makeMatch` は throw する一方、`makeRound` は
  不正固定ペアやコート 0 で `[]` を返す（`guard ... (try? validateFixedPairs()) != nil`）。
  この API 差は呼び出し側のハンドリングに直結するため、明示的にテスト化した。
- **3 対 3 の選出/配置が正しく分岐する**：`teamArrangements` は `assignsSides && count == 2` のときだけ
  ドライブ/バックを割り当て、それ以外は全員 `.any` の 1 通り。`apply` は teamSize に応じて
  味方 (teamSize−1) と対戦 (teamSize) を正しく数える。これらを 6 人 3 対 3 で実測検証した。
- **`NumberedTeam` は初期化時に番号を昇順正規化**：`NumberedTeam([2,1]) == NumberedTeam([1,2])` と
  `Hashable` が成り立つことで、総当たり表の「同じペアの再登場回数」カウントが正しく機能する。

## 影響範囲
- 追加されるのは **テストのみ**。プロダクションコード（`Sources/`）への変更は一切なし＝挙動・API は不変。
- 利用側アプリへの影響なし。CI でのテスト実行時間が約 0.4 秒 → 約 1 秒に増加（許容範囲）。

## 確認内容
- 実施：`swift test` で **92 テスト / 35 スイート全通過**（macOS ホスト上、約 0.96 秒）。
- 未実施：iOS シミュレータ上での実行（ロジックのみ・Foundation 依存ゼロのためホスト結果で十分と判断）。
  コードカバレッジ率の数値計測（`swift test --enable-code-coverage`）は未取得。

## 今後の改善案
- **プロパティベーステスト**の導入余地：出場回数の公平性（差 ≤ 1）やラウンド内の番号非重複は、
  ランダム入力を多数試す形に一般化すると、特定シード依存の穴をさらに塞げる。
- **カバレッジ計測の CI 化**：`swift test --enable-code-coverage` を GitHub Actions に組み込み、
  数値で穴を可視化する（§配布ガイドの CI 整備と併せて）。
- **大人数総当たり表のパフォーマンステスト**：上限引き上げ（近似アルゴリズム）を将来入れる際の
  ベンチ基準として、生成時間を測るテストを別途用意する。
- **同点タイ分布の検証**：タイ・ブレイクの乱数が偏らないこと（多シードでの分布）を統計的に確認するテスト。
