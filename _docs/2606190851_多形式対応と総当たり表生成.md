# 多競技対応（MatchFormat）と総当たり表（乱数）生成の追加

## 概要

SportsPairing を「さまざまなスポーツアプリのペアリング用途」で使えるよう、2 点を拡張した。

1. **試合形式の一般化**：ダブルス専用だったエンジンを、`MatchFormat` で
   シングルス（1 対 1）・ダブルス（2 対 2・サイドあり/なし）・3 対 3 以上に対応させた。
2. **総当たり表（乱数）生成**：背番号ベースで全カードを公平な順に並べる
   `RoundRobinGenerator` を追加（PadeLovers の `RandomNumberTableManager` を一般化）。

v0.1.0 → v0.2.0。テストは 20 → 37 ケースに増加し、全通過。

## 背景

- SportsPairing は複数のスポーツアプリから呼ばれる前提のライブラリ。
  初版はパデル（ダブルス）専用設計だったため、対応競技を広げる必要があった。
- 「乱数」= スポーツのペアリングや総当たりを **数字（背番号）で管理** する仕組み。
  PadeLovers の乱数表機能に相当し、メンバー未登録の集まりでもその場で公平な対戦順を配れる。
  これをライブラリの中核機能として標準化したい、という要望。

## 原因・課題

- 既存 `Matchmaker` は「4 人を選び → 3 通りに分割 → drive/back を割当」とダブルス固定で、
  シングルスや他形式に対応できなかった。`Team` も `drive`/`back` 固定で人数可変に非対応。
- PadeLovers の `RandomNumberTableManager` は 4〜8 人ダブルス限定・重みハードコード・
  シングルス非対応・乱数シード非対応で、そのままでは再利用できなかった。

## 対応内容

### 1. 試合形式の一般化

- 新規 `MatchFormat`（`teamSize`・`usesSides`、プリセット `.singles` / `.doubles` / `.doublesNoSides`）。
- `Team` をメンバー配列ベースに再設計（`Member { player, side }`）。
  ダブルス互換のため `drive`/`back` と `init(drive:back:)` は computed/便宜 init として維持し、既存 API を壊さない。
- `MatchScorer.matchCost` を任意チームサイズに一般化。
  - 味方重複：各チーム内の全ペア平均（シングルスはペアが無く 0）。
  - 対戦重複：チーム間の全ペア平均。
  - サイド不一致：出場人数で正規化。
- `Matchmaker` を一般化。
  - 選出：`selectPlayers(count:)`（= `teamSize × 2`）。固定ペアのユニット化は `teamSize >= 2` のみ。
  - 配置：`teamSplits`（2k 人をサイズ k の 2 チームに分ける全通り。先頭固定で重複排除）+
    `teamArrangements`（ダブルスのみ drive/back の 2 通り、他は全員 .any）。
- `MatchmakingOptions` に `format` を追加（既定 `.doubles`）。

### 2. 総当たり表（乱数）生成

- 新規 `RoundRobinGenerator`、`NumberedMatch` / `NumberedTeam`、`RoundRobinWeights`。
- アルゴリズム（PadeLovers を一般化）：
  1. 1〜N から全チーム（teamSize 人組）を列挙。
  2. 番号が重ならない 2 チームの全組み合わせを候補カードにする。
  3. 各カードに優先度（出場回数×playerBalance + チーム使用×pairRepeat + 顔ぶれ登場×groupRepeat +
     出場済みを後ろへ散らす points 項）を付け、最小から貪欲に並べる。同点はシード乱数。
- `generate` / `generateRounds`（番号が重複しないラウンドへ貪欲分割）/ `maxMatches` 版を提供。
- 重み既定値は PadeLovers 由来（15 / 10 / 10）。

### 3. テスト・ドキュメント

- `FormatTests`（シングルス・サイドなしダブルス）、`RoundRobinTests`（カバレッジ・公平性・
  ラウンド分割・再現性・範囲外エラー）を追加。`swift test` 37 ケース全通過。
- README に「試合形式」「総当たり表（乱数）」節を追加、対応表・ロードマップ・テスト数を更新。

## 設計判断

- **`Team` の後方互換**：member 配列化は破壊的変更になりうるが、`drive`/`back` を computed で
  残すことで既存テスト・API を壊さず一般化。新旧両対応とした。
- **2 つのエンジンを分離**：現場でその都度組む `Matchmaker`（`Player<ID>` ベース）と、
  事前に全カードを並べる `RoundRobinGenerator`（番号ベース）は目的が異なるため別型に。
  前者は状態を持たない逐次生成、後者は全列挙 + 公平順という性質の違いを型で表現した。
- **乱数表の人数上限**：全列挙 + O(M²) 貪欲のため、ダブルス 16 / シングルス 64 を上限とし
  範囲外は明示エラー。大人数の近似対応はロードマップ送り（過剰実装を避ける）。
- **シングルスで固定ペア/partnerRepeat を無視**：味方が存在しないため、設定があっても
  自然に無視する仕様（エラーにせず寛容に）。
- PadeLovers の `flag = playerNumber > 7` による「番号バイアス + 大人数で乱数化」は廃し、
  常にシード乱数でタイ・ブレイクする方式に統一（再現性と一般性を優先）。

## 技術的ポイント

- **チームサイズ非依存のコスト正規化**：ペア数・人数で割ることで、シングルスでもダブルスでも
  各項目が 0〜1 に収まり、重みの意味（相対重要度）が形式を超えて一貫する。
- **重複なしのチーム分割**：2k 人を 2 チームに分ける際、先頭の人を必ずチーム 1 に固定して
  残りから k−1 人を選ぶことで (A|B) と (B|A) の重複を排除（k=2 で 3 通り、k=1 で 1 通り）。
- **総当たりの規模感**：ダブルス N 人のカード数 = N(N−1)(N−2)(N−3)/8。8 人で 210
  （PadeLovers と一致）、16 人で 5460。O(M²) 貪欲のため上限を設けた。
- **ラウンド分割**：並んだカード列を、番号集合が交差しない最初のラウンドへ貪欲に入れる
  （`isDisjoint`）。コート進行表として使える。
- 注意：実装中に SourceKit が同一モジュール内の型を誤検知し続けたが、`swift build`/`swift test` は
  正常。判断はコンパイラ結果を優先。

## 影響範囲

- SportsPairing 内部のみ。`Team` 再設計は後方互換を保ったため既存ダブルステスト 20 件は無修正で全通過。
- PadeLovers 本体・README には今回変更なし（前回作成の README はそのまま有効）。
- 公開 API 追加：`MatchFormat`、`RoundRobinGenerator` 系、`MatchmakingOptions.format`、
  `MatchmakingError.invalidPlayerCount`、`Team.Member` / `members` / 各種 init。

## 確認内容

- 実施：`swift build`（警告・エラーなし）、`swift test`（14 スイート 37 ケース全通過）。
- 既存ダブルス挙動の非回帰を確認（20 ケースが無修正で通過）。
- 未実施：実機統合、大人数（上限超）での性能測定、3 対 3 以上の実用検証（ロジック上は対応）。
